# Step 3 — Coding Practice

### Exercise 1: The Raw Bootstrap (Easy)
* **Objective**: Compile and run a Java program entirely from the command line without an IDE.
* **Difficulty**: Easy
* **Expected Outcome**: Create a file named `Bootstrap.java`. Implement a class that prints out the current Java version (use `System.getProperty("java.version")`). Compile it via `javac Bootstrap.java` and execute it with `java Bootstrap`.
* **Hints**: Ensure the file name matches the public class name exactly.
* **Common Mistakes**: Capitalization errors in the filename or typing `java Bootstrap.class` instead of `java Bootstrap` (do not append `.class` when running).

### Exercise 2: Multi-Class Linking (Challenging)
* **Objective**: Understand class dependencies and manual compilation with classpaths.
* **Difficulty**: Challenging
* **Expected Outcome**: Create two files: `Config.java` (defines a class returning a config string) and `Application.java` (defines the entry point that reads from `Config`). Move `Config.class` to a separate subfolder called `lib/`. Try running `Application` without specifying the classpath, observe the error, and then run it successfully using the classpath flag (`-cp` or `-classpath`).
* **Hints**: Use the command `java -cp .;lib Application` on Windows (semicolon separator) or `java -cp .:lib Application` on Linux/macOS (colon separator).
* **Common Mistakes**: Forgetting to compile both files, or failing to point the execution classpath to the location of the dependency class.

# Step 7 — Engineering Insight

### The JVM Execution Pipeline
Java code is not compiled directly into native machine code (CPU-specific assembly instructions). Instead, it compiles into **bytecode** (a universal, platform-neutral instruction set). 

#### Why it exists:
This design allows Java to fulfill its promise of *"Write Once, Run Anywhere"* (WORA). The JVM translates this bytecode into native machine instructions at runtime on whatever platform it is running (Windows, x86, ARM, Linux, macOS).

#### Trade-offs:
* **Pro**: High portability, memory safety, dynamic class loading.
* **Con**: Warm-up penalty (code runs slower initially until the Just-In-Time compiler compiles hot paths to native machine code) and memory overhead of the JVM process.

#### Senior Approach:
Senior engineers treat compiler diagnostics as immediate feedback. They ensure their code fails fast at compile-time rather than delaying errors to runtime. They structure classes cleanly to keep compilation units decoupled, avoiding monolithic rebuild times in large codebases.

# Step 8 — Open Source Connection

In production servers like **Tomcat** (an open-source servlet container), execution starts in a simple Java main class:
* **Class**: `org.apache.catalina.startup.Bootstrap`
* **Entry Point**: The `main(String[] args)` method in this class instantiates class loaders, sets up the classpath dynamically from your `CATALINA_HOME/lib` folder, loads server configurations, and starts Tomcat. 
* This is exactly the same raw Java execution pipeline you are practicing today, scaled to support millions of enterprise web requests.
