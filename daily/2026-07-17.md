# Step 3 — Coding Practice

### Exercise 1: Block and Scope Boundaries (Easy)
* **Objective**: Explore compiler constraints on block-level variable scopes.
* **Difficulty**: Easy
* **Expected Outcome**: Create a class `ScopePlayground.java`. Write a method containing an `if` block. Declare a variable `int blockVal = 10;` inside the block. Attempt to access `blockVal` outside the `if` block's closing brace. Observe the compiler error. Then, rewrite the code to show the correct way to declare and access a method-scoped variable.
* **Hints**: Scope is defined by curly braces `{}`. A variable declared inside braces is invisible outside them.
* **Common Mistakes**: Expecting variables declared inside loops or conditional blocks to be accessible throughout the method.

### Exercise 2: Stack Frame Simulation (Medium)
* **Objective**: Visualize how the JVM creates independent variable spaces for nested method calls.
* **Difficulty**: Medium
* **Expected Outcome**: Create a class `StackSimulation.java`. Write three nested methods: `levelOne(int x)`, `levelTwo(int x)`, and `levelThree(int x)`. `levelOne` calls `levelTwo(x + 1)`, which calls `levelThree(x + 1)`. At each level, print the value of `x` and the object's system identity (`System.identityHashCode(this)`). Verify that changing `x` inside a nested frame does not affect the value of `x` in the calling frame.
* **Hints**: Each method execution gets its own isolated frame on the stack, holding its own copy of parameter `x`.
* **Common Mistakes**: Conflating primitive parameter reassignments with reference mutations.

# Step 7 — Engineering Insight

### Method Execution & The Call Stack
Every time a Java thread invokes a method, the JVM allocates a **Stack Frame** in the thread's private stack memory.
* **The Frame** contains the method's local variables, the parameter arguments passed by the caller, and execution state pointers.
* **Execution Flow**:
  1. The caller executes a call instruction.
  2. A new frame is pushed to the top of the call stack.
  3. Control transfers to the method bytecode.
  4. When a `return` instruction is reached, the frame is popped off the stack, and execution resumes in the caller's frame.
* **Memory Security**: Because stack frames are strictly separated, local variables in one method cannot bleed into or overwrite variables in another method, guaranteeing local runtime isolation.

# Step 8 — Open Source Connection

In **JUnit 5** or similar testing frameworks:
* The framework uses Java Reflection to locate test methods.
* When a test is run, the engine invokes `method.invoke(testInstance, args)`.
* This causes the JVM to dynamically push a new stack frame for the test method.
* If the test method throws an assertion error, the stack trace is captured by popping frames and recording the stack locations, showing you exactly where the test failed.
