* **Difficulty**: Medium
* **Expected Outcome**: Create a class `StringPerformance.java`. Write two loops that run 10,000 times:
  1. Loop 1: Concatenates a string `"abc"` onto a `String` variable using the `+` operator.
  2. Loop 2: Appends `"abc"` onto a `StringBuilder` instance.
  Measure and print the elapsed time in milliseconds (`System.currentTimeMillis()`) for both loops. Compare the results.

# Step 7 — Engineering Insight

### String Immutability and the String Pool
In Java, `String` objects are immutable. Once created on the heap, their character data cannot be altered. 

#### Why it exists:
1. **Security**: Strings are widely used for file paths, database connections, and network sockets. Immutability prevents values from changing mid-flight.
2. **Caching (String Pool)**: To save memory, JVM maintains a "String Pool" in the heap. If you create two string literals `String s1 = "hello";` and `String s2 = "hello";`, both point to the same memory location in the pool.
3. **Thread Safety**: Immutable objects are inherently thread-safe and can be shared among multiple threads without synchronization.

#### The Cost:
If you modify a string inside a loop (e.g., `s += "test"`), the JVM cannot modify the existing string. It must allocate a new string object, copy the characters over, and update the pointer. In a loop of $N$ iterations, this allocates $N$ discardable garbage objects, causing massive GC pauses. Always use `StringBuilder` for dynamic string constructions.

# Step 8 — Open Source Connection

In libraries like **Jackson** (the JSON parser) or **JUnit** (the testing engine):
* String tokenization and logging use `StringBuilder` or character arrays (`char[]`) directly to assemble output text.
* Because these libraries process millions of strings per second, avoiding string garbage collection overhead is critical for server performance.
