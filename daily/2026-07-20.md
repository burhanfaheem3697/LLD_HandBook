# Step 3 — Coding Practice

### Exercise 1: Checked vs. Unchecked Compilation Check (Easy)
* **Objective**: Differentiate checked and unchecked exceptions at the compilation boundary.
* **Difficulty**: Easy
* **Expected Outcome**: Create a class `ExceptionDemo.java`. Write a method `void triggerChecked()` that throws a custom checked exception `class MyChecked extends Exception {}`. Write another method `void triggerUnchecked()` that throws a custom unchecked exception `class MyUnchecked extends RuntimeException {}`. Try compiling. Observe how the compiler forces a try-catch or a throws declaration on the checked exception, but compiles the unchecked one silently.
* **Hints**: Checked exceptions inherit from `Exception` directly; unchecked exceptions inherit from `RuntimeException`.
* **Common Mistakes**: Expecting the compiler to require handling for `RuntimeException` subclasses.

### Exercise 2: Guard Clauses & Exception Verification (Medium)
* **Objective**: Secure class state boundaries using defensive checks and custom error contracts.
* **Difficulty**: Medium
* **Expected Outcome**: Create a class `InvalidCredentialsException` (checked exception). Create a class `Authenticator.java` with a method `void login(String username, String password) throws InvalidCredentialsException`.
  * Use **guard clauses** at the top of the method: If `username` or `password` is null/empty, throw `IllegalArgumentException` (unchecked).
  * If credentials do not match (e.g., username is not "admin" or password is not "admin123"), throw `InvalidCredentialsException` (checked).
  Create a test runner class using assertions to verify both validation checks and the login failure flows.
* **Hints**: Check bounds/preconditions *before* executing business logic.
* **Common Mistakes**: Swallowing caught exceptions with an empty `catch` block (`catch(Exception e) {}`). Always log or propagate exceptions.

# Step 7 — Engineering Insight

### Checked vs. Unchecked Philosophy
Java is unique in making a hard distinction between Checked and Unchecked Exceptions:
* **Checked Exceptions** represent conditions that a reasonable application should anticipate and recover from (e.g. a database disconnection, network timeout, or file missing). The compiler forces you to declare or handle them.
* **Unchecked Exceptions** (RuntimeExceptions) represent programming bugs or invariant violations (e.g., indexing an array out of bounds, passing a null reference, or division by zero). These indicate the code is broken and should fail fast.

**Senior Rule**: Never swallow exceptions silently. If you catch an exception and do nothing, you leave the application in an indeterminate state, making debugging impossible. If you cannot recover, wrap and rethrow the exception.

# Step 8 — Open Source Connection

In modern frameworks like **Spring Framework**:
* Under the hood, Spring database wrappers convert all checked database exceptions (like JDK's `SQLException`) into unchecked database exceptions (like Spring's `DataAccessException`).
* This decouples business services from low-level database-specific check requirements, keeping code clean while allowing centralized global exception handlers to intercept database failures.
