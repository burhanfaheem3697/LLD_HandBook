# Step 3 — Coding Practice

### Exercise 1: Primitive Type Casting & Overflow (Easy)
* **Objective**: Explore type promotion, implicit casting, and numeric overflow.
* **Difficulty**: Easy
* **Expected Outcome**: Create a class `TypePlayground.java`. Declare a variable of type `byte` at its maximum value (`127`). Add 1 to it. Observe what happens if you cast it back to a `byte`. Write a method that takes a `double` and explicitly casts it to an `int`, printing both.
* **Hints**: `byte b = 127; b = (byte)(b + 1);` causes an overflow. Read about two's complement notation.
* **Common Mistakes**: Expecting `127 + 1` to throw an exception at runtime. In Java, primitive overflow occurs silently.

### Exercise 2: Pass-by-Value Semantics (Medium)
* **Objective**: Prove that Java is strictly pass-by-value for both primitives and references.
* **Difficulty**: Medium
* **Expected Outcome**: Create a class `PassByValueTest.java`. Write two methods:
  1. `modifyPrimitive(int x)`: tries to assign `x = 99`.
  2. `modifyReference(StringBuilder sb)`: tries to call `sb.append(" Modified")` AND tries to reassign `sb = new StringBuilder("New Object")`.
  In your main method, call these and print variables before and after invocation. Analyze why the primitive didn't change, the string was appended, but the object reassignment was lost outside the method.
* **Hints**: Remember: Java passes references *by value*. The method receives a copy of the reference address.
* **Common Mistakes**: Assuming that because objects can be modified inside methods, Java passes objects "by reference". It does not. It passes the reference *value* (address copy).

# Step 7 — Engineering Insight

### Java's Stack vs. Heap Memory Split
Java programs run within a partitioned memory model:
* **The Stack** is fast, local thread-specific memory. Whenever a method is called, a new stack frame is pushed. It stores local primitive variables and object reference addresses. When the method returns, its stack frame is popped instantly, freeing memory.
* **The Heap** is a shared pool of memory where all object instances live. Objects reside here regardless of where they were created. They are managed by the garbage collector (GC), which periodically reclaims objects that no longer have active references pointing to them from the stack.

Understanding this split prevents memory leaks (retaining references on the stack to large heap objects that are no longer needed, preventing garbage collection) and optimizes runtime performance.

# Step 8 — Open Source Connection

In high-performance libraries like **Guava** or **Netty**:
* Custom data structures utilize primitive arrays (e.g., `byte[]` buffers or `int[]` indices) instead of object wrappers (e.g., `List<Byte>` or `List<Integer>`) to avoid **autoboxing**.
* Autoboxing primitives into objects creates unnecessary garbage objects on the heap, increasing GC pressure and degrading performance in high-throughput network applications.
