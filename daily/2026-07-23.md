# Step 3 — Coding Practice

### Exercise 1: Creating Your First JUnit 5 Test (Easy)
* **Objective**: Write and execute a simple unit test checking arithmetic operations.
* **Difficulty**: Easy
* **Expected Outcome**: Create a class `Calculator.java` with a method `int add(int a, int b)`. Create a test class `CalculatorTest.java` containing a `@Test` annotated method. Use `org.junit.jupiter.api.Assertions.assertEquals(5, calculator.add(2, 3))` to verify the math logic.
* **How to Run**: 
  * *If using an IDE (IntelliJ IDEA or VS Code)*: Right-click the `CalculatorTest` class or method and select "Run 'CalculatorTest'". The IDE will download and configure the JUnit libraries automatically.
  * *If compiling manually from console*: You must include the JUnit Jupiter API JAR on your classpath during compilation and execution.
* **Hints**: Ensure your test method has `void` return type and is annotated with `@Test`.
* **Common Mistakes**: Forgetting to import `org.junit.jupiter.api.Test` or using the older JUnit 4 `@Test` annotation (`org.junit.Test`).

### Exercise 2: Verifying Boundary Invariants (Medium)
* **Objective**: Verify exception-throwing logic using JUnit's `assertThrows`.
* **Difficulty**: Medium
* **Expected Outcome**: Create a class `User.java` with a constructor:
  ```java
  public User(String username) {
      if (username == null || username.trim().length() < 3) {
          throw new IllegalArgumentException("Username must be at least 3 characters.");
      }
  }
  ```
  Create a test class `UserTest.java` containing two tests:
  1. `testValidUserCreation()`: Verifies that passing a valid username completes successfully.
  2. `testInvalidUserThrowsException()`: Verifies that passing an invalid username throws `IllegalArgumentException`. Use `assertThrows(IllegalArgumentException.class, () -> new User("ab"));`.
* **Hints**: `assertThrows` accepts a functional interface lambda expression `() -> new User(...)` to execute the code lazily.
* **Common Mistakes**: Writing a standard try-catch block inside a JUnit test to check exceptions. While it works, it is verbose and non-standard; always use `assertThrows`.

# Step 7 — Engineering Insight

### Executable Specifications vs. Post-hoc Verification
In junior developer teams, unit testing is often viewed as a chore completed after writing code to achieve "coverage metrics".

Senior engineers view unit tests as **Executable Specifications**:
* They are living documentation of how the class behaves under valid and invalid inputs.
* Unlike word documents, which drift from reality as code changes, unit tests fail the moment the code deviates from the spec.
* **Design Feedback**: If a class is difficult to unit test (e.g. requires complex setup, mock database layers, or global states), it is a warning sign that the code is tightly coupled. The test suite acts as an immediate design feedback loops.

# Step 8 — Open Source Connection

In libraries like **Google Guava**:
* Every class (e.g. `Iterables`, `Lists`, `Maps`) has corresponding test suites containing thousands of assertions.
* These tests check boundary cases, null rejections, and empty parameters under JUnit.
* When Guava updates its internals for performance, running the test suites verifies that public behaviors remain unchanged, protecting millions of downstream developers from breaking changes.
