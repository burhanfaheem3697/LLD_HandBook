# Notes: Day 1 — Java Program Structure and Execution

## 📌 Key Concepts

* **Static Typing:** Type enforcement before execution.
* **Static Checking:** Bug detection before execution.
* **Dynamic Checking:** Bug detection during execution.

---

## 📖 Core Definitions

### Static Typing

Java is a **statically-typed language**. This means the types of all variables are known at **compile time** (before the program runs), allowing the compiler to deduce the types of all expressions.

> **Example:** If variables `a` and `b` are explicitly declared as `int`, the compiler safely concludes that the expression `a + b` will also yield an `int`.

### The Three Levels of Error Checking

| Checking Level | Description |
| --- | --- |
| **Static Checking** | The bug is found automatically **before** the program even runs. |
| **Dynamic Checking** | The bug is found automatically **when** the code is being executed. |
| **No Checking** | The language doesn’t help you find the error at all. You must catch it yourself or risk silent data corruption. |

---

## 🔍 Deep Dive: Static vs. Dynamic Checking

### 1. What Static Checking Catches

Static checking operates purely on the source code structure and type declarations. It easily identifies:

* **Syntax Errors:** Extra punctuation, unmatched brackets, or spurious words.
*(Note: Even dynamically-typed languages like Python perform basic static syntax checking, such as detecting indentation errors before execution).*
* **Misspelled Names:** Calling `Math.sine(2)` instead of the correct `Math.sin(2)`.
* **Wrong Number of Arguments:** Passing too many parameters, like `Math.sin(30, 20)`.
* **Wrong Argument Types:** Passing invalid data formats, like `Math.sin("30")`.
* **Wrong Return Types:** Trying to execute `return "30";` from a function declared to return an `int`.

### 2. What Dynamic Checking Catches

Dynamic checking relies on the actual values generated at runtime. It halts the program when it encounters:

* **Illegal Argument Values:** The expression `x / y` is only problematic if `y == 0`. The compiler cannot predict what `y` will be, so a divide-by-zero is caught dynamically.
* **Illegal Conversions:** When a runtime value cannot be converted into the target type.
* `Integer.valueOf("hello")` yields a dynamic error because `"hello"` cannot be parsed as a base-10 number.
* `Integer.valueOf("8000000000")` yields a dynamic error because 8 billion falls outside the valid 32-bit range of a Java `int`.


* **Out-of-Range Indices:** Attempting to look up a negative index or an index larger than a sequence's bounds (e.g., arrays or strings).
* **Null Pointer Dereferencing:** Calling a method or accessing a property on a `null` object reference *(Java's equivalent to Python's `None`)*.

## ⚠️ Pitfalls & Lessons Learned

### Case 1: Silent Integer Overflow

```java
int big = 200000; // 200,000
big = big * big;  // Expected: 40,000,000,000 (40 Billion)

```

* **The Problem:** A standard Java 32-bit `int` cannot hold values greater than $2^{31} - 1$ (roughly 2 billion).
* **The Danger:** This is an example of **No Checking**. Java does not catch integer overflows statically or dynamically. It quietly wraps around and produces a completely incorrect negative or truncated value.
* **Industry Context:** Modern languages like *Swift* treat overflow as a dynamic error and halt execution. High-level languages like *Python* and *Ruby* bypass this issue entirely by automatically upgrading primitives to unlimited-precision integers under the hood.

### Case 2: Intentional vs. Accidental Integer Division

```java
double probability = 1 / 5; // Result: 0.0

```

* **The Problem:** The operator `/` is heavily overloaded. Because both operands (`1` and `5`) are literal integers, Java forces **integer division**.
* **The Danger:** The fractional part is quietly truncated, resulting in `0`. Assigning it to a `double` simply turns `0` into `0.0`.
* **The Fix:** At least one operand must be a floating-point literal: `double probability = 1.0 / 5;` (yields `0.2`).

### Case 3: Floating-Point Division by Zero

```java
double sum = 7;
double n = 0;
double average = sum / n; // Result: Infinity

```

* **The Problem:** While integer division by zero throws an explicit dynamic `ArithmeticException`, floating-point division using `double` handles zeroes differently.
* **The Danger:** Instead of crashing, a `double` division by zero evaluates to a special IEEE 754 standard constant: `Double.POSITIVE_INFINITY`.
* **The Takeaway:** If this code is trying to compute a meaningful mathematical average, receiving an unhandled infinity value down the line will likely break your application's logic silently.

# 📋 Questions To Revisit: Java Core & JVM Execution

## Q1: Cross-Platform Execution

**If you compile a class on Windows using `javac`, can you run the resulting `.class` file on a Linux server without recompiling? Why?**

### 💡 The Verdict

**Yes.** Java class files are entirely platform-independent by design.

### ⚙️ How It Works

`javac` does not compile source code down to machine code specific to Windows, macOS, or Linux. Instead, it targets **Java bytecode**—an intermediate, platform-neutral instruction format defined by the JVM specification. This bytecode is captured inside the `.class` file.

The responsibilities are clearly split:

* **`javac` (Compile Time):** Transforms Source (`.java`) $\rightarrow$ Platform-Independent Bytecode (`.class`).
* **`java` / JVM (Run Time):** Transforms Bytecode $\rightarrow$ Native machine instructions optimized on-the-fly for the exact OS/CPU it is running on.

This architecture forms the backbone of Java's **"Write Once, Run Anywhere" (WORA)** promise.

### ⚠️ Critical Edge Cases & Caveats

1. **JVM Version Compatibility:** Bytecode versions are tied to the JDK version used during compilation. A `.class` file compiled via JDK 21 will crash with an `UnsupportedClassVersionError` if run on a Java 8 JVM. Backward compatibility works, but forward compatibility does not.
2. **Native Dependencies:** If your application uses Java Native Interface (JNI) to trigger native OS binaries (`.dll` on Windows, `.so` on Linux), the Java bytecode remains portable, but the underlying native binaries will break across platforms.
3. **Environment Assumptions:** Hardcoded Windows file paths (`C:\projects\data`), line-ending styles (`\r\n`), or file-system case insensitivity can introduce logic bugs on Linux, even if the bytecode executes perfectly.
4. **Architecture-Specific JVMs:** While your bytecode is universal, the JVM *itself* is not. You must install a Linux-specific JVM distribution (e.g., Linux x86_64 or ARM) on the server to run the bytecode.

## Q2: Checked Exceptions vs. Runtime Errors

**What is the difference between a `ClassNotFoundException` (checked exception) and a `NoClassDefFoundError` (runtime error)?**

The fundamental difference lies in **how** the application attempts to load the class. `ClassNotFoundException` occurs during **explicit string-based dynamic loading**, whereas `NoClassDefFoundError` triggers during **implicit linkage** at runtime.

### 📊 Quick Comparison

| Feature | `ClassNotFoundException` | `NoClassDefFoundError` |
| --- | --- | --- |
| **Type** | Checked Exception (`java.lang.Exception`) | Unchecked Error (`java.lang.LinkageError`) |
| **Severity** | Expected & Recoverable (Must be caught/declared) | Fatal & Unrecoverable (Indicates environmental failure) |
| **Trigger** | Explicit lookup via string paths (e.g., `Class.forName()`). | Implicit lookup via class references (e.g., `new MyClass()`). |
| **Root Cause** | The targeted class name string does not exist on the classpath. | The class existed during compilation but vanished or failed to load at runtime. |

### 🔍 Deep Dive

#### 1. `ClassNotFoundException`

Triggered explicitly when using reflection or specific class-loading utilities:

```java
Class.forName("com.mysql.cj.jdbc.Driver");
ClassLoader.loadClass("com.example.MissingClass");

```

* **Why it happens:** Typos in class name strings or a missing dependency JAR file in your project’s build path.
* **Resolution:** Audit string spelling and ensure your runtime classpath includes the missing library.

#### 2. `NoClassDefFoundError`

Triggered implicitly when executing ordinary, well-compiled instructions:

```java
MyClass obj = new MyClass(); // Fails if MyClass.class is missing at runtime

```

* **Why it happens:** A JAR file present during compilation was excluded from the final deployment build, or a class failed its **Static Initialization** (e.g., throwing an `ExceptionInInitializerError`). Subsequent attempts to access that broken class trigger a `NoClassDefFoundError`.
* **Resolution:** Verify your build tool (Maven/Gradle) dependencies are correctly packaged into your deployment unit. If it stems from a static failure, analyze earlier application logs for the initial root exception.

> **💡 Real-World Analogy**
> * **`ClassNotFoundException`:** Like calling directory assistance for a number that doesn't exist. You get an immediate, expected "Number not found" message you must gracefully handle.
> * **`NoClassDefFoundError`:** Like clicking a contact on your speed-dial list who was active yesterday, only to discover their line was disconnected today. The routine action fails unexpectedly mid-task.

## Q3: The `main` Method Signatures

**Why does the `main` method have to be `public` and `static`? What would happen if it were private or instance-based?**

The signature is strictly defined because the JVM acts as an external runtime framework that must kick-start your program without assuming anything about your custom code structure.

### 🛠️ Break Down of Requirements

#### 1. Why must it be `public`?

* **Reason:** The JVM initiates execution from outside your specific class package. If `main` is marked `private`, `protected`, or package-private, the JVM's execution engine lacks the access rights to view or invoke it.
* **Deviation Outcome:** Code compiles. At runtime, the JVM halts and throws a `NoSuchMethodError` indicating the entry point is missing or inaccessible.

#### 2. Why must it be `static`?

* **Reason:** When an application boots up, **zero objects reside in heap memory**. If `main` were an instance method, the JVM would face a paradox: it would need to instantiate an instance of your class to run `main`, but wouldn't know which constructor to pick or what arguments to pass. Making it `static` allows the JVM to execute `ClassName.main()` directly without initial object setup.
* **Deviation Outcome:** Code compiles. Running it results in a clear launch-time error: `Error: Main method is not static in class...`.

### 📋 Runtime Behaviors At a Glance

| Signature Adjustment | Compiles? | Runs? | Runtime Exception Encountered |
| --- | --- | --- | --- |
| `private static void main` | **Yes** | **No** | `Error: Main method not found in class...` |
| `public void main` | **Yes** | **No** | `Error: Main method is not static in class...` |
| `public static int main` | **Yes** | **No** | `Error: Main method must return a value of type void...` |

> **☕ Modern Java Note (Java 21+):** Recent versions introduce *Implicitly Declared Classes and Unnamed Main Methods* as a preview feature, allowing simpler signatures like `void main()`. This serves as compiler-driven syntactic sugar designed for beginners; the under-the-hood JVM platform requirements for large-scale production applications still rest firmly on the standard `public static` entry point.

## Q4: Class Loading Mechanics

**When we execute a program using `java MyApp`, does class loading happen all at once at startup or dynamically on-demand?**

Java adopts a **dynamic, on-demand** approach (commonly called **Lazy Loading**). The JVM refuses to load every compiled `.class` file or third-party dependency into memory upfront.

### ⏱️ The Lifecycle Steps

1. **The Startup Phase:** The JVM prepares core system bootstrap classes (like `java.lang.Object`) and targets the entry class (`MyApp`) strictly to locate and trigger the `main` method.
2. **The On-Demand Phase:** Additional classes are parsed, linked, and initialized **only** when code execution hits one of these runtime triggers for the first time:
* **Object Instantiation:** Invoking the `new` keyword (e.g., `new PaymentProcessor()`).
* **Static Access:** Referencing static parameters or utilities (e.g., `Math.max()` or `Config.ENV`).
* **Reflection:** Calling programmatic lookups like `Class.forName()`.
* **Inheritance:** Loading a child class automatically forces the JVM to evaluate its parent superclass first.



### 🛡️ Why This Architecture Rules

* **Snappy Bootstrapping:** Massive projects utilizing thousands of classes start up instantly because the bulk of the dependency graph is ignored until needed.
* **Lean Memory Profiles:** Niche execution blocks (e.g., an `AdminReportGenerator` triggered once a month) don't consume memory allocations for standard everyday users.

## Q5: Entry Class Isolation

**How does the JVM know which class to execute if there are multiple classes in the same directory?**

The JVM does not search or scan directory trees for candidate files. It relies exclusively on the **fully qualified class name** passed explicitly inside your launch command, navigating via the **Classpath**.

### 🗺️ The Path Discovery Strategy

#### Scenario A: Flat Directory Structuring

If you possess a folder with dozens of unrelated class binaries (`User.class`, `Product.class`, `MainApp.class`) and run:

```bash
java MainApp

```

You provide a explicit instruction. The JVM checks the designated Classpath (which defaults to your current directory `.`), appends `.class` to your argument, and targets `MainApp.class` directly while paying no attention to surrounding files.

#### Scenario B: Package-Driven Structures

In professional projects, classes live inside structured packages. Consider this layout:

```text
myproject/
└── com/
    └── shop/
        ├── MainApp.class
        └── Order.class

```

To start the program, navigate to the root class path directory (`myproject/`) and call the entry target using its full package-qualified name:

```bash
java com.shop.MainApp

```

The JVM parses the dot-notation separators into native directory paths (`com/shop/MainApp.class`) to track down your precise starting point.

#### Scenario C: Executable JAR Bundles

When packaging code inside automated JAR archives, execution shifts to:

```bash
java -jar AppBundle.jar

```

To find the entry target here, the JVM instantly inspects the internal text manifest located at `META-INF/MANIFEST.MF`. It reads the explicit `Main-Class` property to identify the correct starting class:

```text
Manifest-Version: 1.0
Main-Class: com.shop.MainApp

```
