## 📽️ Step 1 — Visual Modeling & UML Specifications

**Lecture referenced:** *UML 2.0 Class Diagrams Tutorial* (Derek Banas)

### Class Representation Anatomy

A standard class is modeled using a vertical **3-compartment box**:

1. **Top** — Class Name
2. **Middle** — Attributes / Fields *(State)*
3. **Bottom** — Methods / Operations *(Behavior)*

### Access Modifier Notation

| Symbol | Modifier | Meaning |
|:---:|---|---|
| `+` | **Public** | Visible everywhere. The JVM relies on this access to reach the system application entry point from an external context. |
| `-` | **Private** | Sealed completely inside the class. |
| `#` | **Protected** | Accessible to the declaring class and its subclasses. |
| `~` | **Package-Private** *(default)* | Restricted strictly to classes within the same package — ideal for encapsulating engine logic and exposing only a lean public contract. |

## 📐 Step 2 — The Object Relationship Spectrum

Tracking how classes interact dictates your system architecture. Relationships scale from the loosest (transient) to the tightest (bound lifecycles):

```text
[ Loose Coupling ] ────────────────────────────────────────────► [ Tight Coupling ]
     Dependency               Association             Aggregation           Composition
 (Method Parameter)       (Long-Term Peers)      (Loose Collection)     (Exclusive Death Grip)
       ..>                       ────►                  ◇────                ◆────
```

### 1. Dependency `..>`
- **Definition:** A transient, temporary "uses" relationship. Class A relies on Class B for a brief duration, typically inside a specific method block.
- **Code Footprint:** Class B is passed into a method as a parameter, or instantiated inside a block as a temporary local variable.
- **UML Rule:** Dotted line with an open arrow, pointing **to** the supplier class (`Application ..> Config`).

### 2. Association `────►`
- **Definition:** A structural, long-term connection between independent peer objects. Neither owns the other — they simply interact over their lifecycles (e.g. a `Doctor` and a `Patient`).
- **Code Footprint:** Class A holds a persistent reference to Class B as an instance field.
- **UML Rule:** Solid line pointing to the referenced class.

### 3. Aggregation `◇────`
- **Definition:** An asymmetric **"Whole-to-Part"** relationship where the parts can survive outside the container (e.g. a `Library` holding independent `Book` objects).
- **Code Footprint:** Child objects are created *outside* the container and injected via constructor or setter.
- **Lifetime Test:** Container destroyed → contained objects survive.
- **UML Rule:** Solid line with a **hollow diamond** on the "Whole" side.

### 4. Composition `◆────`
- **Definition:** A strict **"Whole-to-Part"** relationship with exclusive ownership and absolute lifecycle dependency (e.g. an `Engine` and its inner `Valves`).
- **Code Footprint:** The parent "gives birth" to the child internally (typically inside its constructor); the child is rarely exposed externally.
- **Lifetime Test:** Container dies → children are destroyed with it.
- **UML Rule:** Solid line with a **filled/solid diamond** on the "Whole" side.

## 📖 Step 3 — Core Reading Insights: *"A Trip to Objectville"*

**Source:** *Head First Java*, Chapter 2 (pp. 27–48)

### 🏛️ Paradigm Shift: Procedural vs. Object-Oriented Design

- **Procedural Design** relies heavily on global/passive data structures separated from the isolated functions that manipulate them. As a system scales, a small change to a data structure's format can cause a cascading domino effect across dozens of statically-coupled functions.
- **Object-Oriented Design** packages state (data) and behavior (methods) together inside self-contained objects, using encapsulation to shield internal values and prevent global data contamination.

### 🔒 Architectural Depth: Bad Getters & "Tell, Don't Ask"

Simply making fields `private` and exposing them via public getters can leak internal types, reproducing the same fragile static coupling found in procedural design. If a getter returns a custom internal domain object, any change to that type breaks every external consumer.

> **The Solution:** Abandon the "Ask" approach. Instead of pulling an object's internal data out to compute externally, **tell** the object what business behavior to execute.

**Stable Interfaces** — if you must expose data, map internal objects to deeply stable, universally understood contracts (primitives or standard system classes) to shield consumers from internal refactoring.

### 🧠 Blueprints vs. The Heap

| Concept | Description |
|---|---|
| **The Class** | A compile-time blueprint — rules, specs, types, structure. Takes up **no heap space**. |
| **The Object** | A live runtime entity, allocated memory on the **JVM Heap**. |
| **The Reference Variable** | A pointer on the thread stack holding an object's heap address. Nullifying it cuts the connection, making the object eligible for **Garbage Collection**. |

## 💻 Step 4 — Coding Exercise Hardening & Toolchain Mechanics

### 🏗️ File Infrastructure & Setup

Classes were arranged across isolated conceptual directories to test manual package building and path resolution from a clean terminal workspace:

```text
workspace/
├── Config.java         ← Returns system greeting string
├── Application.java    ← Contains main entry point method
├── AppTester.java      ← Formulates built-in verification logic
└── bin/                ← Dedicated destination target for compiled bytecode
```

### 🛠️ Terminal Execution Steps & Advanced CLI Quirks

#### 1. Compile with Destination Management

The `-d` (destination) compiler flag redirects compiled artifacts into a target folder without cluttering the workspace:

```bash
javac -d bin/ Config.java Application.java AppTester.java
```

**CLI Argument Flexibility** — `javac` uses a flexible argument parser: flags can appear before or after the source filenames, as long as each flag is immediately followed by its own path argument. Both of these work identically:

```bash
javac -d ./bin MyClass.java
javac MyClass.java -d ./bin
```

#### 2. Verify Automation via the Native Assertion Engine

Java skips assertion execution by default to optimize production runtime. To turn our code into a primitive test runner, pass the explicit **Enable Assertions flag (`-ea`)** with a classpath definition:

```bash
# Unix-based Systems (Linux/macOS)
java -ea -cp bin:. AppTester

# Windows CMD / PowerShell
java -ea -cp "bin;." AppTester
```

If a conditional check inside the test script fails, the JVM raises a clean, localized `AssertionError` straight to standard output.

> **Shell Quoting Mechanics** — a single classpath path doesn't strictly need quotes (`java -cp bin Main`), but multi-path definitions separated by delimiters (`;` or `:`) **must** be wrapped in double quotes. This forces the shell to treat the segment as one literal string instead of misinterpreting the delimiter as a shell operator.

## 🚀 Step 5 — Engineering Insight: Compile-Time vs. Runtime

| Characteristic | Compile-Time (`javac`) | Runtime (`java`) |
|---|---|---|
| **Enforcer** | The Java Compiler | The JVM Dynamic ClassLoader |
| **Objective** | Verifies type safety, valid method definitions, syntax, and contract matches | Locates, links, and dynamically loads bytecode into heap memory for execution |
| **Artifact** | Raw `.java` source & structural headers | Compiled `.class` files / JARs on the active Classpath |

### ⚠️ Architectural Importance & The Stale Bytecode Trap

This strict division lets us compile against static interface contracts while swapping implementation bytecode at runtime. But because the JVM uses **lazy loading**, running an out-of-sync build — where an internal dependency's method signature changed without recompiling its consumer — will still pass startup validation. The failure only surfaces **mid-execution**, at the exact line of invocation, as a `java.lang.NoSuchMethodError`.

## 🧠 Step 6 — Core Knowledge Retention Check

<details>
<summary><b>1. If you compile a program and delete a class that's never instantiated, will the JVM crash at startup?</b></summary>

<br>

**No.** The JVM uses a lazy, on-demand class-loading policy — it initializes minimal entry parameters at boot and only performs a classpath lookup when an instruction explicitly reaches out to a missing reference type.
</details>

<details>
<summary><b>2. How does the <code>-ea</code> flag alter bytecode configuration?</b></summary>

<br>

**It doesn't alter bytecode at all.** The compiler *always* generates assertion logic inside the compiled `.class` file. The `-ea` flag simply tells the JVM runtime to evaluate those bytes instead of skipping over them.
</details>

<details>
<summary><b>3. Why is <code>Application</code> → <code>Config</code> a dependency (<code>..></code>) rather than an association (<code>───►</code>)?</b></summary>

<br>

Because `Application` doesn't preserve a persistent reference to `Config` in a class-level instance field. The lookup is localized entirely within a single method's short-lived scope, making it a transient connection.
</details>

<details>
<summary><b>4. Why are package-private (default) classes ideal for decoupling dependencies?</b></summary>

<br>

They restrict access exclusively to elements inside the same package. By keeping internal engine classes package-private and exposing only a lean `public` interface, external classes can't compile against your internals — letting you refactor or delete internal code anytime without breaking consumers.
</details>
