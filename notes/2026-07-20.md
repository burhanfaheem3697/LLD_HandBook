## 1. Warm-Up & Foundations Review

### Quick Recall Concepts

| Concept | Key Takeaway |
|---|---|
| **Reference Variables by Value** | Java copies the *reference itself* by value. Both caller and method point to the same heap object — mutations are visible globally, but reassigning the local parameter (`sb = new StringBuilder()`) does **not** affect the caller's variable. |
| **Encapsulating Mutable Collections** | `final` or removing a setter is **not** enough. Real protection requires either a **defensive copy** (`new ArrayList<>(internalList)`) or an **unmodifiable wrapper** (`Collections.unmodifiableList(...)`, `List.copyOf(...)`), which throws `UnsupportedOperationException` on mutation attempts. |
| **UML Sequence Diagram Lifelines** | When Method A calls Method B, A's activation bar stays open (paused) while a new bar opens on B's lifeline. B's bar closes on return, then control resumes on A's bar. |
| **Stack Frame Memory** | Frames are pushed on invocation and popped instantly on `return` or an unhandled exception, via the thread's stack pointer. The **Garbage Collector manages heap only** — it has no role in stack frame lifecycle. |
| **JUnit 5 Test Isolation** | JUnit 5 creates a **brand-new instance** of the test class for every single `@Test` method, guaranteeing no shared state leaks between tests. |

### ✅ Warm-Up Verification Snippet

```java
import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.assertEquals;

class ConfigurationTest {

    void updateConfig(StringBuilder sb) {
        sb.append("-PROD"); // State mutation on the shared heap object
    }

    @Test
    void testUpdateConfigMutatesOriginalObject() {
        StringBuilder sb = new StringBuilder("APP");
        updateConfig(sb);

        // Formally asserts that the heap object mutated downstream
        assertEquals("APP-PROD", sb.toString());
    }
}
```

## 2. Core Exceptional Paradigms

### 🔒 Checked Exceptions
Represent **expected, recoverable, external** conditions (e.g., `IOException`, `FileNotFoundException`). The compiler **strictly enforces** handling — callers must `try-catch` or declare `throws`.

### ⚡ Unchecked Exceptions (`RuntimeException`)
Represent **internal bugs or invariant violations** (e.g., `NullPointerException`, `IllegalArgumentException`). The compiler **ignores** enforcement entirely — they're meant to **fail fast** so developers fix the underlying defect rather than paper over it.

### 📜 The Software Engineering Contract

| Element | Role |
|---|---|
| `requires` clause | Preconditions — demands placed on the client |
| `effects` clause | Postconditions — guarantees, including declared exceptions |
| Special values (`null`, `-1`) | ⚠️ Hazardous — easy to forget checking, causes silent failures |
| Exceptions | Violently halt execution, making failures **unignorable** |

## 3. Coding Practice & Implementation Blueprints

### Exercise 1 — Compilation Bounds Checking

```java
public class ExceptionDemo {
    // Custom Exception Types
    class MyChecked extends Exception {}
    class MyUnchecked extends RuntimeException {}

    // Compiler forces a try-catch or a 'throws' declaration!
    public void triggerChecked() throws MyChecked {
        throw new MyChecked();
    }

    // Compiles silently without any mandatory declaration or try-catch
    public void triggerUnchecked() {
        throw new MyUnchecked();
    }
}
```

### Exercise 2 — Guard Clauses & Automated Exception Validation

```java
// Custom Checked Exception representing a legal business alternative path
class InvalidCredentialsException extends Exception {
    public InvalidCredentialsException(String message) {
        super(message);
    }
}

class Authenticator {
    public void login(String username, String password) throws InvalidCredentialsException {
        // 1. Guard Clauses: Fail-Fast early boundary verification (Unchecked)
        if (username == null || username.trim().isEmpty() ||
            password == null || password.trim().isEmpty()) {
            throw new IllegalArgumentException("Credentials cannot be null or empty.");
        }

        // 2. Core Business Logic & Checked Exception Contract Execution
        if (!"admin".equals(username) || !"admin123".equals(password)) {
            throw new InvalidCredentialsException("Invalid username or password configuration.");
        }

        System.out.println("Login successful.");
    }
}
```

### Manual Testing Execution Architecture

Without a full test framework, use an `AssertionError` trap to catch false passes:

```java
public class TestRunner {
    public static void main(String[] args) {
        Authenticator auth = new Authenticator();

        // Test Case: Guard clause interception verification
        try {
            auth.login("", "admin123");
            // If the line below executes, our code failed to throw the expected exception!
            throw new AssertionError("Test Failed: IllegalArgumentException was not thrown!");
        } catch (IllegalArgumentException e) {
            System.out.println("Test Passed: Guard clause correctly rejected empty username.");
        }
    }
}
```

## 4. Deep Dive: Open-Source Connection & The Spring Strategy

### ⚠️ The Architectural Problem with Vanilla JDBC

Standard JDBC throws a **checked** `java.sql.SQLException`, forcing `throws SQLException` across every layer (Repository → Service → Controller). This causes:

1. **Leaky Abstractions** — Business logic gets structurally tied to SQL; swapping to NoSQL requires rewriting method signatures.
2. **Boilerplate Pollution** — Unrecoverable infra crashes clutter code with useless handling statements.

### 🌱 Spring's Solution: Exception Translation

Spring intercepts low-level, vendor-specific checked exceptions and translates them into a centralized **unchecked** hierarchy: `org.springframework.dao.DataAccessException`.

```
[Low-Level Checked Exception]  --->  Intercepted by Spring  --->  [Centralized Unchecked Exception Hierarchy]
     java.sql.SQLException                                          DataAccessException (RuntimeException)
```

### Preserving Fallbacks via Rich Runtime Hierarchies

`DataAccessException` uses a broad, polymorphic tree — Spring maps precise vendor error codes to clean subclasses, so developers can still catch exactly what they need:

```java
public class UserService {
    private UserRepository userRepository;

    public void registerUser(User user) {
        try {
            userRepository.save(user);
        } catch (DuplicateKeyException e) {
            // Intentional Business Fallback: Resolve duplicate username anomaly
            generateAlternativeUsernameAndSave(user);
        } catch (QueryTimeoutException e) {
            // Infrastructure Fallback: Route the operation to a safe retry queue
            sendToRetryQueue(user);
        }
        // Note: Any unrecoverable structural failures (like connection dead)
        // bubble up automatically without mandatory throws boilerplate.
    }
}
```

> **Core Philosophy:** A `try-catch` performs identically whether the exception is checked or unchecked. Spring simply removes the *mandatory compiler requirement*, letting engineers target recoverable cases explicitly while infrastructure crashes clean out the thread predictably.

## 5. End-of-Day Reflection

<details>
<summary><strong>Q1. Why does Java compile unchecked exceptions silently, but checks others at compile-time?</strong></summary>

Unchecked exceptions represent internal programming bugs (e.g., `NullPointerException`) — the compiler expects developers to *fix the source code*, not build fallback logic around it. Checked exceptions represent external, unpredictable environmental hazards (missing hardware, lost connections) that code can't control but must anticipate, so the compiler enforces a structural recovery path.

</details>

<details>
<summary><strong>Q2. If a method throws an exception, what happens to the variables in its stack frame?</strong></summary>

Execution halts immediately, and the JVM begins **unwinding the call stack** to find a matching `catch` block. The method's stack frame is popped off instantly, destroying its local variables — they do not persist while an outer method handles the exception.

</details>

<details>
<summary><strong>Q3. Does <code>finally</code> execute if <code>catch</code> contains a <code>return</code>? What if <code>catch</code> throws?</strong></summary>

Yes — `finally` is an unyielding contract. If `catch` triggers a `return`, the JVM pauses it, fully runs `finally`, then completes the return. If `catch` throws a new exception, the JVM pauses that exception, runs `finally` completely, then lets the exception continue bubbling up.

> ⚠️ **Warning:** If `finally` itself throws or returns, it will **overwrite and swallow** whatever was coming from the `catch` block.

</details>

<details>
<summary><strong>Q4. Why is throwing a generic <code>RuntimeException</code> discouraged?</strong></summary>

It obscures intent, hurts readability, and forces callers into wide catch nets (`catch (RuntimeException e)`) that can accidentally swallow unrelated bugs (indexing errors, null fields) that should have crashed the app to expose bad logic.

</details>

<details>
<summary><strong>Q5. When should you choose a Checked Exception over a RuntimeException for an API contract?</strong></summary>

Choose **checked** when failure stems from external, unpredictable realities (I/O drops, API timeouts, filesystem restrictions) where the caller has both the ability and operational need for a graceful fallback. Choose **unchecked** when the exception signals structural misuse of the API (e.g., invalid input boundaries).

</details>

## 🔑 Key Terms Glossary

| Term | Definition |
|---|---|
| `throws` | Declares a checked exception a method may propagate |
| `finally` | Block guaranteed to execute regardless of exception/return |
| `DataAccessException` | Spring's unchecked wrapper around vendor-specific data errors |
| Guard Clause | Early validation check that fails fast on invalid input |
| Defensive Copy | Returning a new collection instance to prevent external mutation |
| Stack Unwinding | JVM process of popping frames to locate a matching `catch` |
