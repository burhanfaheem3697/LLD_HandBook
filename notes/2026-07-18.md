## 1. Warm-Up & Execution Context Recall

### 🧱 Stack Frame Mechanics & Recursion

| Concept | Detail |
|---|---|
| **Isolation of Execution** | Every method invocation pushes a distinct stack frame onto the JVM call stack. |
| **Recursive Safety** | Each recursive call gets its own isolated stack frame with a private local variable array — no overwriting between depths. |
| **Variable Lifecycles** | When a method completes, its stack frame is popped instantly. Local/parameter variables are destroyed — but if a parameter held an object reference, only the *reference* dies. The object itself lives on the heap until the Garbage Collector reclaims it. |

### 🔍 Scope Shadows & Testing Frameworks

- **Variable Shadowing** — A local variable or parameter with the same name as an instance variable *shadows* it. Access the instance variable explicitly via `this.variableName`.
- **Block Nesting Boundaries** — Variables declared inside an `if` block or loop are invisible outside that block. An inner block also can't redeclare a name that already exists in the outer method scope.
- **JUnit 5 Stack Frame Capturing** — A failed assertion throws `AssertionFailedError` (extends `java.lang.Throwable`). The JVM snapshots the call stack via `Throwable.fillInStackTrace()`, letting JUnit report the exact file and line of failure.

## 2. Compile-Time Polymorphism (Method Overloading)

### The Rule of the Method Signature

A method's identity in Java is defined **strictly** by its signature:

```
Method Signature = Method Name + Parameter List (Type, Order, Count)
```

**Ignored** during signature resolution:

- ❌ Return Type
- ❌ Access Modifiers (`public`, `private`, `protected`)
- ❌ Scope Modifiers (`static` vs. non-static)
- ❌ Thrown Exceptions

### ⚠️ Critical Architectural Constraints

<details>
<summary><strong>Why return types alone can't overload a method</strong></summary>

Two methods with identical names and parameter lists but different return types (e.g. `int calculate()` vs `double calculate()`) create ambiguity. A bare call like `calculate();` gives the compiler no destination type to infer from — it can't choose.
</details>

<details>
<summary><strong>The static vs. non-static clash</strong></summary>

You can't declare a `static` method and an instance method with the same parameter signature in one class. Since `static` isn't part of the signature, the compiler flags it as a duplicate declaration.
</details>

<details>
<summary><strong>The ambiguity of <code>null</code></strong></summary>

Passing a bare `null` to overloaded methods like `calculateAverage(int[] values)` vs `calculateAverage(double[] values)` is ambiguous — `null` is valid for both. Fix it with an explicit cast:

```java
calculateAverage((int[]) null);
```
</details>

## 3. Variable-Arity Parameters (Varargs)

### Operational Rules

1. ✅ A varargs parameter (`Type... name`) must be the **final** parameter in the list.
2. ✅ A method can have **at most one** varargs parameter.

### 🔧 Under the Hood

The compiler translates `int... numbers` into a plain array: `int[] numbers`.

If a non-final varargs position were allowed —

```java
// ❌ Not legal Java
printNumber(int... numbers, int x)
```

— then a call like `printNumber(1, 2, 3, 4);` would be structurally ambiguous. Forcing varargs to the end lets the compiler parse standard arguments first and safely pack the trailing values into the array.

## 4. Master Insight: Static Binding vs. Dynamic Binding

The key distinction between the **Java Compiler (`javac`)** and the **JVM**.

| | Static Binding | Dynamic Binding |
|---|---|---|
| **Resolved by** | Compiler, before execution | JVM, at runtime |
| **Applies to** | Overloaded, `static`, `private`, `final` methods | Overridden methods (polymorphism) |
| **Mechanism** | Compiler bakes the exact target address into `.class` bytecode | Compiler emits `invokevirtual`; JVM checks the object's **Virtual Method Table (VMT)** at runtime |
| **Speed** | No runtime lookup — hardcoded path | Runtime lookup based on actual heap object |

### 🤔 Why the Compiler Refuses to Bind Overridden Code Statically

Even with `Parent obj = new Child();`, the compiler is forced into dynamic binding because:

1. **Variable Re-assignment Capacity** — `obj` could be reassigned later (`obj = new AnotherChild();`), and the compiler can't predict runtime state.
2. **Separate Compilation Philosophy** — Java compiles classes independently. Hardcoding would break polymorphism if a subclass changed without recompiling the caller.

> **🔑 The `final` Exception**
> If a method or class is marked `final`, no subclass can ever override it. Only then can the compiler safely optimize and resolve the binding **statically**.

## 5. Real-World Connections

**`java.util.Arrays`** leans heavily on compile-time overloading — `Arrays.sort()` alone exposes 15+ overloads (`sort(int[])`, `sort(double[])`, `sort(char[])`, ...). One intuitive API name, many type-optimized implementations under the hood.

## 6. Mistakes & Gotchas to Avoid

- 🚫 Trying to overload by changing only return type, access modifier, or `static` status.
- 🚫 Misplacing a varargs parameter before regular arguments, or declaring more than one.
- 🚫 Passing a naked `null` into overloaded object-reference methods without an explicit cast.
