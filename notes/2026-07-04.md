## 📹 2 · Video Lecture Notes

**Source:** *Stanford CS106A — Lecture 14: Memory*

### Memory Partition Architecture

- **Stack Frame Lifecycle** — Every thread owns an isolated, fast-access stack. Calling a method pushes a new frame holding parameters and local primitives. When the method returns, the frame is popped instantly — local scope cleanup is automatic and immediate.
- **Heap Pool Allocation** — The heap is one shared, global memory pool for *all* object instances, regardless of where they were created. Object lifetime is decoupled from any single stack frame and is managed by the **Garbage Collector (GC)**.

### Reference Semantics

- A reference variable on the stack doesn't hold object data — it holds an **address** pointing to the object on the heap.
- Multiple reference variables can point to the *same* address, like several remote controls aimed at one TV.

## 📚 3 · Book Reading Notes

**Source:** *Head First Java (3rd Edition)* — Ch. 3, "Know Your Variables"

### Primitive Data Types & Precision Boundaries

| Primitive | Size | Range / Properties |
|---|---|---|
| `boolean` | JVM-dependent | `true` / `false` |
| `byte` | 8 bits | `-128` → `127` |
| `char` | 16 bits | `0` → `65,535` (unsigned Unicode) |
| `short` | 16 bits | `-32,768` → `32,767` |
| `int` | 32 bits | `-2,147,483,648` → `2,147,483,647` |
| `float` | 32 bits | IEEE 754 single-precision |
| `long` | 64 bits | `-9.22×10¹⁸` → `9.22×10¹⁸` |
| `double` | 64 bits | IEEE 754 double-precision |

### Memory Management & Reference Scope

> 🎛️ **The Remote Control Analogy**
> Primitives store raw values; objects store *references* (pointers). Copying an object reference copies the "remote control," never the underlying "TV."

**Garbage Collection Eligibility** — An object becomes eligible for GC the instant it becomes *unreachable*: no active reference, in any stack frame, still points to it.

## 💻 4 · Coding Practice & Architectural Verification

### 🔹 Exercise 1 — Primitive Casting & Overflow (`TypePlayground.java`)

Demonstrates silent arithmetic wraparound via **Two's Complement** and precision loss from narrowing casts.

```java
package handbook.phase00.p00m01l02;

public class TypePlayground {
    public static void main(String[] args) {
        // 1. Silent Byte Overflow Analysis
        byte maxByte = 127;
        // Binary of 127: 01111111 (MSB 0 = positive)

        byte overflowedByte = (byte) (maxByte + 1);
        // Addition pushes a 1 into the sign bit: 10000000
        // In Two's Complement, 10000000 == -128

        System.out.println("Max Byte Value: " + maxByte);
        System.out.println("Overflowed Byte Value (127 + 1): " + overflowedByte);

        // 2. Explicit Narrowing Cast: double -> int
        double piPrecision = 3.14159;
        int truncatedPi = (int) piPrecision; // fractional bits discarded

        System.out.println("Original Double: " + piPrecision);
        System.out.println("Casted Integer: " + truncatedPi);
    }
}
```

### 🔹 Exercise 2 — Strict Pass-by-Value Verification (`PassByValueTest.java`)

Proves Java is *always* pass-by-value — even for object references, only the **pointer copy** is passed.

```java
package handbook.phase00.p00m01l02;

public class PassByValueTest {
    public static void main(String[] args) {
        int primitiveVal = 10;
        StringBuilder referenceObj = new StringBuilder("Original");

        System.out.println("=== BEFORE METHOD EXECUTIONS ===");
        System.out.println("Primitive Variable Address Value: " + primitiveVal);
        System.out.println("StringBuilder Internal Content   : " + referenceObj);

        modifyPrimitive(primitiveVal);
        modifyReference(referenceObj);

        System.out.println("\n=== AFTER METHOD EXECUTIONS ===");
        System.out.println("Primitive Variable (Stays 10)    : " + primitiveVal);
        System.out.println("StringBuilder Content (Mutated)  : " + referenceObj);
    }

    public static void modifyPrimitive(int x) {
        x = 99; // only the local copy changes
    }

    public static void modifyReference(StringBuilder sb) {
        sb.append(" Modified");              // mutates the shared heap object
        sb = new StringBuilder("New Object"); // rebinds only the local copy
    }
}
```

| Call | What Actually Happens | Visible Outside? |
|---|---|---|
| `modifyPrimitive(x)` | Local copy of `int` reassigned | ❌ No |
| `sb.append(...)` | Mutates the shared heap object | ✅ Yes |
| `sb = new StringBuilder(...)` | Rebinds only the local pointer copy | ❌ No |

## ⚙️ 5 · Engineering Insight: Stack vs. Heap Memory Split

```text
┌─────────────────────────────┬──────────────────────────────┐
│         STACK MEMORY         │          HEAP MEMORY         │
│   (thread-local, fast-scoped)│    (globally shared pool)    │
├─────────────────────────────┼──────────────────────────────┤
│ [main]                       │                              │
│   primitiveVal = 10          │   ┌────────────────────┐     │
│   referenceObj ─────────────►├──►│  StringBuilder     │     │
│                               │   │  "Original"        │     │
│ [modifyReference]             │   └────────────────────┘     │
│   sb (address copy) ─────────┤       ▲                       │
│   sb reassigned ──────────────┼──┐   │ mutated → "Modified"  │
│                               │  │   │                       │
│                               │  ▼                            │
│                               │  ┌────────────────────┐      │
│                               │  │  StringBuilder     │      │
│                               │  │  "New Object"       │      │
│                               │  └────────────────────┘      │
└─────────────────────────────┴──────────────────────────────┘
```

| | Stack | Heap |
|---|---|---|
| **Scope** | Thread-local | Globally shared |
| **Speed** | Extremely fast (LIFO push/pop) | Slower, requires tracking |
| **Lifetime** | Ends when method returns | Managed by GC |
| **Holds** | Primitives + reference addresses | Actual object instances |
| **Leak risk** | None — auto-popped | High if references are held too long |

> ⚠️ **LLD Significance:** Holding onto stack references to large, stale heap objects blocks the GC from reclaiming that memory — a classic root cause of leaks in custom caches, collections, and long-lived listeners.

## 🔌 6 · Open Source Connection — High-Performance Collections

In throughput-critical libraries like **Netty** and **Google Guava**, default OOP collections are often too slow.

**The problem — Autoboxing overhead:**
Using `List<Integer>` or `List<Byte>` wraps every primitive in a full heap object (`Integer`, `Byte`), costing **3–4× the memory** per element and flooding the heap with transient garbage → frequent GC pauses.

**The fix — Raw primitive arrays:**
High-performance engines use `int[]`, `byte[]`, etc., storing values contiguously. This bypasses autoboxing entirely, keeps GC pressure near zero, and dramatically improves L1/L2 CPU cache locality.

## 👁️ 7 · End-of-Day Reflection & Technical Resolution

<table>
<tr><td><b>1. Literal Suffix Requirement</b></td></tr>
<tr><td>

**Q:** Why does `float f = 3.14;` fail to compile while `double d = 3.14;` succeeds?

**A:** A decimal literal defaults to `double` (64-bit). Assigning it to a 32-bit `float` is a narrowing conversion, so the compiler blocks it. Fix: `float f = 3.14f;`

</td></tr>
<tr><td><b>2. Lifespan of Unreferenced Heap Objects</b></td></tr>
<tr><td>

**Q:** What happens when the only stack reference to a heap object goes out of scope?

**A:** The object becomes unreachable immediately. It sits idle on the heap until the GC's sweep reclaims it and returns the memory to the pool.

</td></tr>
<tr><td><b>3. Integer Division & Truncation</b></td></tr>
<tr><td>

**Q:** Why does `System.out.println(10 / 3);` print `3`, not `3.333...`?

**A:** Both operands default to `int`, so the result is computed as `int` — the fractional part is truncated, not rounded. Fix: `10.0 / 3`

</td></tr>
<tr><td><b>4. Modifying Heap State via Pass-by-Value</b></td></tr>
<tr><td>

**Q:** If Java is pass-by-value, how can a method mutate an object's state?

**A:** The *value* passed is the reference (address). The method's copy and the caller's original point to the same heap object, so mutations via that address (`.append()`, field sets) are visible everywhere.

</td></tr>
<tr><td><b>5. Local Variables vs. Heap Field Defaults</b></td></tr>
<tr><td>

**Q:** Are local primitives auto-initialized with default values?

**A:** **No** — trick question. Local variables hold garbage until explicitly assigned; using one unassigned is a compile error (`variable might not have been initialized`). Default values (`0`, `false`, etc.) apply only to **instance fields** and **static variables**.

</td></tr>
</table>

## 📋 8 · Things That Clicked Today

- ✅ Java is **strictly pass-by-value** everywhere — object arguments pass a *copy of the address*, never the variable itself.
- ✅ Primitive overflow wraps silently via **Two's Complement** math — no exception is thrown, so boundary validation must be proactive in LLD value objects.
- ✅ Stack frames are deterministic and instant; heap pools are dynamic, shared, and reclaimed asynchronously by the GC.
