## ⚡ TL;DR — 60 Second Recap

- A **unit test** checks one small piece of logic in isolation — no real DB, network, or file I/O.
- Test methods: annotated `@Test`, always `void`, always **zero arguments**.
- `@BeforeEach` / `@AfterEach` run around *every* test; `@BeforeAll` / `@AfterAll` run *once* for the *whole class*.
- `assertEquals` → compares **values** (`.equals()`), gives rich diffs. `assertTrue` → compares **booleans**, gives vague errors. Prefer `assertEquals` when possible.
- `assertThrows` needs a **lambda** because Java evaluates arguments *before* calling a method — without a lambda, the exception would escape before JUnit can catch it.
- Keep unit tests fast by **injecting dependencies** (interfaces) instead of hardcoding real DB/network/email implementations — swap in mocks/fakes for tests.
- If a class is painful to test (needs 10 mocks), that's not a testing problem — it's a **design smell**.

## 1. Core Concepts & Rules

| Concept | Explanation |
|---|---|
| **Unit Test Purpose** | Isolates and verifies a single "unit" (method/class) works correctly, independent of external systems like databases, APIs, or networks. |
| **Must be `@Test`** | Marks the method so JUnit's reflection engine discovers and runs it. |
| **Must return `void`** | JUnit determines pass/fail by catching thrown `AssertionError`s — not by inspecting a return value. |
| **Must take no arguments** | JUnit invokes test methods via reflection and has no mechanism to supply parameters (unless using parameterized tests). |
| **Fresh instance per test** | By default, JUnit uses the `PER_METHOD` lifecycle — a **new instance** of the test class is created for every `@Test`, preventing state from leaking between tests. |

> 💡 **Why this matters:** These constraints aren't arbitrary Java rules — they exist so JUnit can safely discover, isolate, and run hundreds of tests automatically without you writing any boilerplate "test runner" code yourself.

## 2. The Testing Lifecycle

Think of the lifecycle as **"setup → test → cleanup"**, repeated per test, with optional one-time bookends for the whole class.

| Annotation | Runs When | Typical Use |
|---|---|---|
| `@BeforeAll` | **Once**, before any test in the class | Expensive one-time setup (e.g., starting a shared resource). Must be `static` unless using `PER_CLASS` lifecycle. |
| `@BeforeEach` | Before **every** `@Test` method | Re-initialize fresh objects/fixtures so each test starts clean. |
| `@Test` | The actual test | Contains the assertion logic. |
| `@AfterEach` | After **every** `@Test` method | Cleanup (e.g., clearing in-memory state). |
| `@AfterAll` | **Once**, after all tests finish | Tear down shared/expensive resources. |

**Mental model:**
```
@BeforeAll        ─────────────────────────────┐  (runs once)
   @BeforeEach → @Test 1 → @AfterEach           │
   @BeforeEach → @Test 2 → @AfterEach           │  (repeats per test)
   @BeforeEach → @Test 3 → @AfterEach           │
@AfterAll         ─────────────────────────────┘  (runs once)
```

## 3. Assertions & Failure Diagnostics

| Assertion | What it Checks | Failure Message Quality |
|---|---|---|
| `assertEquals(expected, actual)` | Value equality via `.equals()` | ✅ Rich — e.g. `Expected: "Alice", Actual: "Bob"` |
| `assertTrue(condition)` | A boolean expression | ⚠️ Vague — e.g. just `Expected true, but was false` |

> ⚠️ **Common pitfall:** Avoid `assertTrue(a == b)` for object comparisons — `==` checks **memory reference**, not content, and even when it does fail, the error message won't tell you *what* was actually wrong. Use `assertEquals(a, b)` instead for meaningful diffs.

## 5. CLI Compilation & Execution Guide

### 🤔 Why do we need the JAR *twice*?

| Stage | Command Flag | Purpose |
|---|---|---|
| **Compilation** | `javac -cp` | Acts like a **dictionary** — tells the compiler that types like `@Test` and `assertEquals` exist, without this it won't compile. |
| **Execution** | `java -jar` | `junit-console.jar` is an **executable engine** with its own `main` method. It scans `.class` files via reflection and actually invokes the `@Test` methods. |

### 💻 Terminal Commands (JUnit 6 Console Standalone)

```cmd
:: 1. Compile both application and test class
javac -cp "junit-console.jar;." User.java UserTest.java

:: 2. Execute all tests in directory
java -jar junit-console.jar execute --classpath . --scan-classpath

:: 3. Execute a specific test class
java -jar junit-console.jar execute --classpath . --select-class UserTest

:: 4. Execute a specific test method
java -jar junit-console.jar execute --classpath . --select-method UserTest#testInvalidUserThrowsException
```

> 🗂️ **Quick recall trick:** *"Compile needs the dictionary, run needs the engine."*

## 6. Deep Dive: Why `assertThrows` Needs Lambdas

**The core Java rule:** method arguments are evaluated **before** the method itself is called.

| Approach | What Happens | Result |
|---|---|---|
| ❌ `assertThrows(Exception.class, new User("ab"));` | `new User("ab")` runs **immediately**, *before* `assertThrows` even starts. | The exception escapes unhandled → crashes the test process. |
| ✅ `assertThrows(Exception.class, () -> new User("ab"));` | The code is wrapped in a **lambda** (an `Executable` "recipe") that isn't run yet. | `assertThrows` opens the recipe itself, inside its **own internal `try-catch`**, safely catching and verifying the exception. |

> 💡 **Analogy:** A lambda here is like handing someone a **sealed envelope with instructions** instead of doing the action yourself. `assertThrows` is the only one allowed to open the envelope — inside a safety net (`try-catch`) — so if something explodes, it's contained.

## 7. Decoupling Dependencies (DB, Network, Email)

Unit tests must remain **fast** and **isolated** — no real infrastructure calls allowed.

**Key techniques:**

- **Dependency Injection (DI):** Pass *interfaces* (e.g. `UserRepository`, `EmailService`) into a class's constructor instead of instantiating real implementations inside the method. This lets tests swap in fakes.
- **Mocks / Fakes:** During testing, inject fake/mock implementations that behave correctly but run instantly in-memory — no real network or DB round trip.

### 🧭 Testing Taxonomy

| Test Type | Speed | Tools | What It Tests |
|---|---|---|---|
| **Unit Tests** | Milliseconds | Mocks / Fakes | Logic in isolation |
| **Integration Tests** | Seconds | Testcontainers | Real interactions with DB/APIs |
| **End-to-End Tests** | Minutes | Staging Environment | The full system workflow |

> 🗂️ **Quick recall trick:** *"Unit = fake & fast. Integration = real & slower. E2E = everything & slowest."*

## 8. Avoiding the "Mocking Nightmare"

**Symptom:** Writing a unit test requires mocking an endless chain of nested objects.

| Step | Fix |
|---|---|
| **1. Recognize the smell** | This usually means the class violates the **Law of Demeter** (talking to "strangers" too many levels deep) or is doing too much. Refactor long object chains. |
| **2. Don't mock value objects** | Never mock simple data objects/POJOs like `User` or `Address` — just instantiate real ones directly: `new User("Alice")`. |
| **3. Use Test Data Builders** | Create helper functions (e.g. `TestDataBuilder.createDummyUserWithPosts(5)`) to eliminate duplicated, verbose test setup code. |

## 9. Engineering Insights & Philosophy

> 🧠 **"Tests as Executable Specifications & Design Feedback"**
>
> Senior engineers treat unit tests as **living documentation** that never goes stale (unlike comments or wikis). More importantly: **test pain is design feedback.** If a class is painful and tedious to unit test, that pain is a warning sign — your design is likely:
> - Tightly coupled
> - Doing too much (violating Single Responsibility)
> - Missing key abstractions

> 🗂️ **Quick recall trick:** *"If the test is ugly, the code is ugly."*

## 10. Open-Source Connection

- Massive open-source projects like **Google Guava** rely on thousands of automated JUnit tests running in continuous integration (CI) pipelines.
- Every public API, boundary case, and bug fix in Guava is paired with a regression test — protecting the millions of downstream projects that depend on it from silent breakage.

## 11. End-of-Day Reflection Q&A

| # | Question | Core Answer |
|---|---|---|
| **1** | **State Isolation** | `@BeforeEach` re-initializes test fixtures before *every* test method, preventing state mutations from leaking across tests. |
| **2** | **Assertions & Diagnostics** | `assertEquals` tests value equality (`.equals()`) and gives detailed expected/actual failure messages; `assertTrue` tests boolean expressions and gives vague error reports. |
| **3** | **Exception Testing** | `assertThrows` uses lambdas to defer execution until JUnit can wrap the call inside its own `try-catch` block. |
| **4** | **Method Signatures** | Test methods must be `void` because JUnit detects results through pass/fail execution and exceptions, ignoring return values. |
| **5** | **Design Feedback** | Difficulty writing isolated tests indicates high coupling or low cohesion, signaling it's time to refactor architecture. |
