## 📖 Reading Notes

**Source:** *Head First Java (3rd Edition)* — Ch. 5 *"Extra-Strength Methods"* & Ch. 6 *"Using the Java Library"*

### Standard vs. Enhanced `for` Loops
- **Standard `for`** → use when you need structural modification or index tracking.
- **Enhanced `for`** → use for pure, readable sequential access.

### Autoboxing Overhead
Generic collections like `ArrayList` can't hold primitives (`int`, `char`) directly — the compiler forces them to store **object wrappers** (`Integer`, `Character`) instead. In high-throughput loops, this adds pointer overhead and extra garbage collection pressure.

## 🛠️ Coding Practice

### Exercise 1: Exception Boundaries (`IndexPlayground.java`)

Java throws different exceptions depending on whether you're touching a raw array or an API class:

| Access | Exception Thrown | Why |
|---|---|---|
| `arr[23]` (primitive array) | `ArrayIndexOutOfBoundsException` | Low-level, built into the JVM's memory access checks |
| `list.get(5)` (dynamic collection) | `IndexOutOfBoundsException` | Thrown explicitly by the class after checking its internal `size` field |

> **Inheritance note:** `ArrayIndexOutOfBoundsException` is a subclass of `IndexOutOfBoundsException`.

### Exercise 2: String vs. StringBuilder (`StringPerformance.java`)

| Approach | Time (10,000 iterations) | Why |
|---|---|---|
| `str += "abc"` | ~87ms | Strings are **immutable** — every concatenation allocates a new instance and copies all prior characters. Overall complexity: **O(N²)** |
| `sb.append("abc")` | ~2ms | `StringBuilder` mutates an internal char array **in place**, only resizing when full. Amortized complexity: **O(N)** |

## 💡 Engineering Insight: String Immutability & The Pool

Java strings can never change after creation — a design choice with three big payoffs:

1. **🔒 Security** — prevents tampering with sensitive runtime values (DB URLs, credentials, file paths).
2. **🗄️ The String Pool** — a dedicated heap cache; literal declarations like `String s = "hello"` reuse existing matching entries to save memory.
3. **🧵 Thread Safety** — no data races, since multiple threads can safely share the same immutable string with zero synchronization overhead.

## 🔌 Open Source Connection

High-performance Java libraries lean hard on these principles:

- **Jackson** (JSON parsing) and **JUnit** (test compilation) minimize garbage collection overhead from string manipulation.
- They work directly with `char[]` arrays or pre-sized `StringBuilder` buffers to process millions of operations per second while keeping GC pauses minimal.

## 🧠 End-of-Day Reflection

### 1. The Explicit Constructor Trap — `new String("hello")`
Using `new` forces the JVM to **skip the String Pool**, even if an identical literal already exists there. A brand-new duplicate object is created on the general heap, throwing away the pool's reuse benefits.

### 2. Layout Divergence: `int[]` vs. `ArrayList<Integer>`

| | `int[] arr = new int[100]` | `ArrayList<Integer> list = new ArrayList<>(100)` |
|---|---|---|
| Storage | One contiguous block of primitive values | Array of references/pointers |
| Objects | None — raw values | 100 separate `Integer` objects scattered in memory |
| Cache locality | ✅ Excellent | ❌ Poor — reference overhead |

### 3. Modifications Inside Enhanced `for` Loops

- **Structural changes** (`add()`, `remove()`) → break the `Iterator`'s internal tracking → throws `ConcurrentModificationException`.
- **Value changes** (`for (int val : arr) { val = 5; }`) → only mutates a **local loop-scoped copy**. The source array is untouched.

### 4. Resizing Mechanics of `StringBuilder`

When capacity runs out, `StringBuilder` grows using:

$$\text{New Capacity} = (\text{Old Capacity} \times 2) + 2$$

Existing data is copied into the new buffer. Since this doubling happens exponentially less often as the buffer grows, the amortized runtime stays at **O(N)**.

### 5. Type Erasure Constraints on Generics

- **The constraint:** Generics are compiled down to `Object` via **Type Erasure**.
- **The impact:** Primitives don't extend `Object`, so `ArrayList<int>` is invalid. Java works around this with **autoboxing** — automatically wrapping primitives into objects — at the cost of extra heap pressure.
