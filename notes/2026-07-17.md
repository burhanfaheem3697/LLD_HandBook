## ⏱️ Warm-up: Quick Recall Concepts

| Concept | Key Takeaway |
|---|---|
| `ConcurrentModificationException` | For-each uses an `Iterator`; `iterator.next()` compares `modCount` vs. `expectedModCount`. A structural change mid-loop desyncs them and trips Java's fail-fast check. |
| Shallow vs. Deep Copy | Shallow copy duplicates structure only (shared object references); deep copy duplicates every element into new heap objects. |
| Defensive Copying | Protects a collection's *structure* from outside mutation — but a shallow copy still leaks access to mutable elements inside it. Deep copy is required for full encapsulation. |
| UML Sequence Loops | A `loop` fragment repeats while its **Guard Condition** `[condition]` (top-left of the box) holds true. |
| Case-Insensitive Matching | Normalizing case before comparison avoids duplicate-entry bugs and matches input regardless of typing style. |

### 🧩 Raw Array Substring Filtering

Filtering without collections means handling `null`s and resizing manually:

```java
private static String[] getFilteredTasks(String[] tasks, String query) {
    if (tasks == null || query == null) {
        return new String[0];
    }

    String[] tempFiltered = new String[tasks.length];
    int matchCount = 0;
    String loweredQuery = query.toLowerCase();

    for (String task : tasks) {
        // Null protection + case-insensitive substring matching
        if (task != null && task.toLowerCase().contains(loweredQuery)) {
            tempFiltered[matchCount++] = task; // Retains original task casing
        }
    }

    // Truncate array to exact match size to eliminate trailing nulls
    String[] result = new String[matchCount];
    System.arraycopy(tempFiltered, 0, result, 0, matchCount);

    return result;
}
```

> 💡 **Why it matters:** without truncation, the result array would be padded with trailing `null`s, causing NPEs downstream.

## 📑 Specifications as a Behavioral Contract

*(MIT 6.031)*

A specification is a formal **contract** between the **client** (caller) and the **implementer** (writer) — it isolates components so implementations can evolve without breaking callers.

| Clause | Who It Binds | Meaning |
|---|---|---|
| **`requires` (preconditions)** | Client | What must be true *before* the call. Broken precondition → implementer owes nothing (undefined behavior is fair game). |
| **`effects` (postconditions)** | Implementer | What's guaranteed *after* the call — but only if preconditions held. |

**Spec strength:** a specification is *stronger* when it asks less of the client (looser `requires`) and promises more to the client (tighter `effects`). A stronger spec can always safely replace a weaker one.

> 🔒 **Mutability rule:** if `effects` doesn't say a parameter is mutated, the implementer must **not** mutate it.

## 🛡️ Checked vs. Unchecked Exceptions

| Feature | Checked Exceptions | Unchecked Exceptions |
|---|---|---|
| **Compiler Enforcement** | Strict — must be handled | None — compiler ignores them |
| **How to Handle** | `try-catch` or `throws` in signature | Optional; bubbles up automatically |
| **Class Hierarchy** | Extends `java.lang.Exception` (not `RuntimeException`) | Extends `java.lang.RuntimeException` |
| **Philosophy** | Anticipated, recoverable, external conditions | Programming bugs / invalid assumptions |
| **Examples** | `IOException`, `FileNotFoundException`, `SQLException` | `NullPointerException`, `ArrayIndexOutOfBoundsException`, `IllegalArgumentException`, `ConcurrentModificationException` |

**Checked** exceptions guard against failures outside the code's control (file I/O, networks, databases) — the compiler forces the caller to catch or propagate them.

**Unchecked** exceptions signal logic errors. They're designed to **fail fast** during development rather than be silently recovered from.

## 📚 JVM Architectural Blueprints

*(Head First Java)*

### Stack vs. Heap

- **Call Stack** — per-thread, holds **Stack Frames** (local vars, primitives, execution pointer). Frames pop instantly on `return` or unhandled exception.
- **Heap** — shared space for **all objects**. Objects live as long as a reference chain reaches them; the Garbage Collector reclaims the rest.

### Pass-by-Value — Always

Java is **strictly pass-by-value**, even for objects:

| Type | What's Copied | Effect |
|---|---|---|
| Primitive | The literal value | Changes stay local to the callee frame |
| Object reference | The memory address (pointer) | Reassigning the parameter is local-only, but mutating the object (`obj.setValue(5)`) changes the shared heap instance |

### Object Creation Chain

- **Constructor chaining:** `super()` must run first so the foundation is initialized before subclass fields.
- ⚠️ **Polymorphic constructor danger:** never call an overridable method from a constructor — the subclass override can fire *before* the subclass's own fields are initialized, producing `null`/uninitialized-state bugs.

## 🔒 Scope Isolation: Java vs. C++

Java **forbids** method-level variable shadowing; C++ allows it.

```cpp
// C++ — Legal Shadowing
void process() {
    int x = 10;
    if (true) { int x = 5; } // ✅ shadows outer x
}
```

```java
// Java — Compilation Error
public void process() {
    int x = 10;
    if (true) { int x = 5; } // ❌ 'x' is already defined in method
}
```

- **Why:** Java trades block-level isolation for readability, avoiding accidental re-declaration bugs.
- **The one exception:** instance fields *can* be shadowed by constructor/setter parameters (`this.name = name`), since `this` disambiguates unambiguously.

## 💻 Coding Exercise: Stack Frame Simulation

Uses `System.identityHashCode(obj)` to inspect the raw memory-based hash, bypassing any overridden `.hashCode()`.

```java
public class StackSimulation {
    public static void main(String[] args) {
        int x = 5;
        StackSimulationHelper helper = new StackSimulationHelper();
        helper.levelOne(x);
    }
}

class StackSimulationHelper {
    public void levelOne(int x) {
        System.out.println("x : " + x + " | Instance ID: " + System.identityHashCode(this));
        levelTwo(x + 1);
    }

    public void levelTwo(int x) {
        System.out.println("x : " + x + " | Instance ID: " + System.identityHashCode(this));
        levelThree(x + 1);
    }

    public void levelThree(int x) {
        System.out.println("x : " + x + " | Instance ID: " + System.identityHashCode(this));
    }
}
```

**Output:**

```text
x : 5 | Instance ID: 140435067
x : 6 | Instance ID: 140435067
x : 7 | Instance ID: 140435067
```

**Analysis:** the changing `x` values prove each call gets an independent stack frame with its own primitive slot. The **identical** instance ID proves all three calls run against the same single heap object.

## ⚙️ Engineering Insight & Open Source Connection

- **Memory allocation efficiency:** stack allocation is just a pointer bump (contiguous, per-thread, cheap). Heap allocation is comparatively expensive — fragmented memory forces the JVM to search for free blocks and track them.
- **JUnit 5 hook:** test frameworks use Java Reflection (`method.invoke(testInstance, args)`) to trigger tests, which pushes a fresh stack frame. On assertion failure, the framework pops frames while capturing execution pointers to build the stack trace you see in test reports.

## 🏁 End-of-Day Reflection

1. **Call Stack Lifecycle** — frames push on call, pop instantly on return, destroying local state.
2. **References** — object passing copies the pointer only; no heap duplication occurs.
3. **Variable Lifespans** — instance vars live with their heap object (GC-managed); locals live with their stack frame.
4. **Shadowing Ban** — disallowed within a single Java method to prevent reassignment mistakes.
5. **Stack > Heap (Performance)** — sequential pointer movement beats non-contiguous heap searches.
